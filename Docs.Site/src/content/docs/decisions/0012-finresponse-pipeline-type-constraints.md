## 맥락과 문제

Mediator 파이프라인(로깅, 유효성 검증, 트랜잭션 등)이 모든 요청/응답 타입에 무차별적으로 적용되면, 파이프라인 내부에서 응답 타입을 `object`로 캐스팅하거나 리플렉션에 의존해야 한다. 이로 인해 런타임 에러가 발생하고, 특정 파이프라인이 어떤 응답 타입을 기대하는지 컴파일 타임에 알 수 없다.

`FinResponse<T>`를 사용하는 파이프라인에서 타입 안전하게 성공/실패를 판별하고, 에러 정보에 접근하며, 새로운 응답을 생성할 수 있는 제약 체계가 필요하다.

## 검토한 옵션

- **옵션 1**: `object` 캐스팅
- **옵션 2**: 단일 인터페이스로 모든 기능 통합
- **옵션 3**: IFinResponse 4단계 인터페이스 계층 + `where` 제약
- **옵션 4**: 리플렉션 기반 동적 디스패치

## 결정

**옵션 3: IFinResponse 4단계 인터페이스 계층 + `where` 제약을 채택한다.**

인터페이스를 4단계로 분리하여 각 파이프라인이 필요한 최소 조건만 제약으로 요구하도록 한다.

1. **IFinResponse (마커)**: 응답이 FinResponse 계열임을 표시. 가장 느슨한 제약으로, 모든 FinResponse 파이프라인의 기본 진입점.
2. **IFinResponseCovariant (공변)**: `IsSucc`/`IsFail` 성공 여부 판별. 로깅 파이프라인 등 읽기 전용 접근에 사용.
3. **IFinResponseFactory (CRTP 팩토리)**: `Fail<T>(error)` 등 새로운 응답 인스턴스 생성. 유효성 검증 파이프라인에서 실패 응답을 만들 때 사용.
4. **IFinResponseErrorAccessor (에러 접근)**: 실패 시 에러 상세 정보 접근. 에러 분류, 관측성 파이프라인에서 사용.

각 파이프라인의 `where TResponse :` 제약에 해당 단계의 인터페이스만 지정하므로, 파이프라인이 요구하는 최소 계약이 타입 시그니처에 드러난다.

### 결과

- **긍정적**: 파이프라인의 타입 제약이 컴파일 타임에 강제되어 런타임 캐스팅 에러가 사라진다. 각 파이프라인이 필요한 최소 인터페이스만 요구하므로 결합도가 낮다. 새 파이프라인 추가 시 어떤 수준의 접근이 필요한지 인터페이스 계층에서 명확히 선택할 수 있다. 85개 테스트로 검증 완료.
- **부정적**: 4단계 인터페이스 계층이 초기 학습 곡선을 높인다. 인터페이스가 많아 IDE의 자동완성 목록이 길어질 수 있다.

### 확인

- 85개 단위 테스트가 모든 파이프라인과 인터페이스 조합을 검증한다.
- 잘못된 `where` 제약을 사용할 경우 컴파일 에러가 발생하는지 확인한다.
- 새로운 파이프라인을 추가할 때 적절한 인터페이스 단계를 선택하는 가이드를 제공한다.

## 옵션별 장단점

### 옵션 1: object 캐스팅

- **장점**: 구현이 가장 단순하다. 인터페이스 설계가 불필요하다.
- **단점**: 런타임 `InvalidCastException`이 발생할 수 있다. 타입 안전성이 전혀 없어 리팩토링 시 에러를 사전에 감지할 수 없다. 파이프라인이 어떤 응답 타입을 기대하는지 코드만으로 파악이 불가능하다.

### 옵션 2: 단일 인터페이스로 모든 기능 통합

- **장점**: 인터페이스가 하나이므로 학습이 쉽다. 모든 파이프라인에 동일한 제약을 사용한다.
- **단점**: 로깅 파이프라인처럼 읽기만 필요한 곳에도 팩토리, 에러 접근 등 불필요한 메서드가 노출된다. 인터페이스 분리 원칙(ISP) 위반. 새 기능 추가 시 모든 구현체를 수정해야 한다.

### 옵션 3: IFinResponse 4단계 인터페이스 계층

- **장점**: 인터페이스 분리 원칙을 준수한다. 각 파이프라인이 최소 제약만 요구하여 결합도가 낮다. 공변성을 활용한 읽기 전용 접근이 가능하다. CRTP 패턴으로 타입 안전한 팩토리를 제공한다. 컴파일 타임 검증으로 런타임 에러를 방지한다.
- **단점**: 4단계 인터페이스 계층의 설계 복잡도가 있다. 새 팀원에게 CRTP 패턴 설명이 필요하다. 인터페이스 수가 많아 문서화 부담이 있다.

### 옵션 4: 리플렉션 기반 동적 디스패치

- **장점**: 인터페이스 설계 없이 런타임에 유연하게 처리할 수 있다. 기존 타입을 수정할 필요가 없다.
- **단점**: 리플렉션으로 인한 성능 저하가 매 요청마다 발생한다. 컴파일 타임 검증이 없다. 디버깅이 어렵고 스택 트레이스가 복잡하다. Hot path에서 사용하기에 부적합하다.
