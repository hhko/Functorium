## 맥락과 문제

운영 환경에서 비즈니스 오류(예: 유효성 검증 실패, 재고 부족)와 시스템 장애(예: DB 연결 실패, 타임아웃)가 동일한 방식으로 로깅되면 알림 피로가 발생하고, 장애 대응 우선순위를 판단하기 어렵다. 비즈니스 오류는 정상 흐름의 일부이므로 경고 수준이면 충분하지만, 시스템 장애는 즉각적인 대응이 필요하다. 또한 벌크 연산에서 발생하는 복합 오류는 개별 오류의 단순 집합 이상의 맥락을 가진다.

OpenTelemetry 3-Pillar(Trace, Metrics, Logs) 전반에서 오류 원인을 일관되게 구분할 수 있는 분류 체계가 필요하다.

## 검토한 옵션

- **옵션 1**: 단일 error.type 문자열
- **옵션 2**: HTTP 상태 코드 기반 분류
- **옵션 3**: Expected/Exceptional/Aggregate 3-Type 자동 분류
- **옵션 4**: 에러 분류 없이 로깅만 수행

## 결정

**옵션 3: Expected/Exceptional/Aggregate 3-Type 자동 분류를 채택한다.**

오류를 3가지 유형으로 자동 분류하고, `error.type`과 `error.code` 필드를 통해 OpenTelemetry 전 계층에서 일관되게 전파한다.

- **Expected (비즈니스 오류)**: 도메인 규칙 위반, 유효성 검증 실패 등 예상 가능한 오류. 경고(Warning) 수준으로 처리하며, 비즈니스 메트릭으로 집계한다.
- **Exceptional (시스템 장애)**: 인프라 장애, 미처리 예외 등 예상하지 못한 오류. 에러(Error) 수준으로 즉시 알림 대상이 된다.
- **Aggregate (복합 오류)**: 벌크 연산에서 발생한 다수의 개별 오류를 하나로 묶어 전체 맥락(성공/실패 비율, 공통 원인)을 제공한다.

파이프라인에서 응답 타입을 검사하여 자동으로 분류하므로, 개발자가 매번 분류를 지정할 필요가 없다.

### 결과

- **긍정적**: 알림 규칙을 오류 유형별로 분리할 수 있어 알림 피로가 감소한다. 대시보드에서 비즈니스 오류 추이와 시스템 장애를 별도 패널로 모니터링할 수 있다. 벌크 연산의 부분 실패를 단일 Aggregate 오류로 파악할 수 있다.
- **부정적**: 새로운 오류 타입을 추가할 때 분류 로직을 함께 갱신해야 한다. 분류 기준에 대한 팀 내 합의가 필요하다.

### 확인

- `error.type` 필드가 Trace Span, Metric Attribute, Structured Log에 동일하게 기록되는지 확인한다.
- Expected 오류가 알림을 트리거하지 않고, Exceptional 오류만 알림이 발생하는지 확인한다.
- Aggregate 오류에 개별 오류 목록과 요약 정보가 포함되는지 확인한다.

## 옵션별 장단점

### 옵션 1: 단일 error.type 문자열

- **장점**: 구현이 단순하다. 추가 인프라가 필요 없다.
- **단점**: 비즈니스 오류와 시스템 장애의 원인 구분이 불가능하다. 알림 규칙을 세분화할 수 없어 알림 피로가 지속된다. 문자열 기반이므로 오타나 불일치가 발생하기 쉽다.

### 옵션 2: HTTP 상태 코드 기반 분류

- **장점**: 널리 알려진 표준이다. 기존 모니터링 도구와 호환이 좋다.
- **단점**: 비즈니스 맥락이 손실된다. 동일한 400 Bad Request가 유효성 검증 실패인지 권한 부족인지 구분할 수 없다. 비-HTTP 통신(gRPC, 메시지 큐)에 적용하기 어렵다. 도메인 오류 코드와 1:1 매핑이 불가능하다.

### 옵션 3: Expected/Exceptional/Aggregate 3-Type 자동 분류

- **장점**: 비즈니스 오류와 시스템 장애를 명확히 구분하여 운영 효율이 향상된다. OpenTelemetry 3-Pillar 전체에서 일관된 분류를 유지한다. 파이프라인에서 자동 분류하므로 개발자 부담이 적다. 벌크 연산의 복합 오류를 체계적으로 처리한다.
- **단점**: 분류 체계를 학습해야 하는 초기 비용이 있다. 경계 케이스(예: 비즈니스 규칙이지만 즉시 대응이 필요한 경우)에서 판단이 모호할 수 있다.

### 옵션 4: 에러 분류 없이 로깅만 수행

- **장점**: 구현 비용이 없다. 기존 코드 변경이 불필요하다.
- **단점**: 모든 오류가 동일하게 취급되어 운영 대응이 비효율적이다. 대시보드와 알림을 원인별로 세분화할 수 없다. 오류 패턴 분석이 사실상 불가능하다.
