## 맥락과 문제

기존 도메인 이벤트 발행기는 `IDomainEventBatchHandler`를 통해 벌크 연산 시 동일 타입의 이벤트를 묶어 처리했다. 그러나 이 구현은 리플렉션 기반 100줄의 디스패치 로직을 포함하고 있어, 디버깅이 어렵고 런타임 에러 추적이 복잡했다. 이벤트 타입별 제네릭 핸들러를 동적으로 찾아 호출하는 과정에서 타입 불일치 시 명확한 에러 메시지가 제공되지 않았다.

복잡한 리플렉션 로직을 제거하면서도 벌크 연산의 이벤트 처리 요구를 충족할 수 있는 단순한 구조가 필요했다.

## 검토한 옵션

- **옵션 1**: IDomainEventBatchHandler 유지 및 리플렉션 최적화
- **옵션 2**: BatchHandler 제거 + 벌크 로직을 Domain Service로 이동
- **옵션 3**: 이벤트 발행 자체를 제거

## 결정

**옵션 2: BatchHandler를 제거하고, 벌크 로직을 Domain Service로 이동한다. (BREAKING CHANGE)**

- `IDomainEventBatchHandler` 인터페이스와 리플렉션 기반 디스패치 로직을 완전히 제거한다.
- 벌크 연산에서 발생하는 이벤트 처리는 명시적인 Domain Service(예: `ProductBulkOperations`)에서 담당한다.
- 도메인 이벤트 발행기는 단일 이벤트 발행에만 집중하여 책임이 명확해진다.
- Domain Service에서 벌크 연산의 결과를 집계하고, 필요한 경우 요약 이벤트를 발행한다.

### 결과

- **긍정적**: 리플렉션 100줄이 제거되어 코드가 단순해지고 디버깅이 쉬워졌다. Domain Service에 벌크 로직이 명시적으로 드러나 비즈니스 의도가 코드에서 직접 읽힌다. 이벤트 발행기의 단일 책임이 확보되었다. 타입 안전성이 향상되었다.
- **부정적**: BREAKING CHANGE로 기존 BatchHandler 구현체를 사용하는 코드가 모두 Domain Service로 마이그레이션해야 한다. 벌크 연산마다 전용 Domain Service를 만들어야 하는 보일러플레이트가 발생할 수 있다.

### 확인

- 기존 BatchHandler를 사용하던 모든 코드가 Domain Service로 마이그레이션되었는지 확인한다.
- 벌크 연산에서 개별 이벤트가 누락 없이 발행되는지 통합 테스트로 검증한다.
- Domain Service의 벌크 로직이 트랜잭션 경계 내에서 올바르게 동작하는지 확인한다.

## 옵션별 장단점

### 옵션 1: IDomainEventBatchHandler 유지 및 리플렉션 최적화

- **장점**: 기존 코드 변경이 최소화된다. BatchHandler 패턴에 익숙한 팀원이 적응할 필요가 없다. 벌크 이벤트 처리가 프레임워크 수준에서 자동으로 이루어진다.
- **단점**: 리플렉션의 근본적인 복잡도는 해결되지 않는다. 디버깅 어려움이 지속된다. 성능 오버헤드(리플렉션 호출)가 매 벌크 연산마다 발생한다. 타입 불일치 시 런타임 에러가 발생한다.

### 옵션 2: BatchHandler 제거 + Domain Service로 이동

- **장점**: 리플렉션이 완전히 제거되어 코드가 단순하고 디버깅이 쉽다. 벌크 연산의 비즈니스 로직이 Domain Service에 명시적으로 표현된다. 이벤트 발행기의 책임이 단일화된다. 컴파일 타임 타입 안전성이 확보된다.
- **단점**: BREAKING CHANGE로 마이그레이션 비용이 발생한다. 벌크 연산마다 Domain Service를 작성해야 하는 부담이 있다. 프레임워크 수준의 자동화가 줄어든다.

### 옵션 3: 이벤트 발행 자체를 제거

- **장점**: 가장 단순하다. 이벤트 관련 인프라가 전혀 필요 없다.
- **단점**: 도메인 이벤트를 통한 부수 효과(알림, 감사 로그, 캐시 무효화) 처리가 불가능하다. Aggregate 간 느슨한 결합을 포기하게 된다. 일관성 유지 메커니즘이 사라진다.
