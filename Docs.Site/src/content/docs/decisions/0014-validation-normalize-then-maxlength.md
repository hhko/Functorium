## 맥락과 문제

Value Object 생성 시 입력값의 유효성 검증 순서에 따라 결과가 달라진다. 예를 들어 `"  Hello  "`(앞뒤 공백 포함 9자)에 대해 MaxLength(5) 검사를 Trim 전에 수행하면 길이 9로 판단하여 거부하지만, Trim 후에는 `"Hello"`(5자)로 통과한다.

Trim 전에 MaxLength를 검사하면 사용자가 의도한 유효한 입력이 공백 때문에 잘못 거부되며, 반대로 정규화하지 않고 DB에 저장하면 공백이 포함된 데이터가 남아 불일치가 발생한다. 유효성 검증과 정규화의 올바른 순서를 정해야 한다.

## 검토한 옵션

- **옵션 1**: MaxLength 검사 후 Normalize (현재 순서)
- **옵션 2**: Normalize 후 MaxLength 검사
- **옵션 3**: Normalize 미적용 (원본 그대로 저장)
- **옵션 4**: 단일 메서드에서 정규화와 검증을 동시 수행

## 결정

**옵션 2: `ThenNormalize(Trim)` 후 `ThenMaxLength` 순서를 고정한다.**

유효성 검증 파이프라인에서 다음 순서를 강제한다:
1. `ThenNormalize(Trim)` — 입력값을 먼저 정규화(공백 제거 등)
2. `ThenMaxLength(n)` — 정규화된 값에 대해 길이 검증

`CreateFromValidated` 팩토리 메서드는 정규화가 완료된 데이터만 수락하여, 정규화되지 않은 원본 데이터가 Value Object로 생성되는 것을 방지한다.

### 결과

- **긍정적**: 공백 포함 입력의 잘못된 거부가 사라진다. DB에 저장되는 데이터가 항상 정규화된 형태이므로 검색, 비교 시 일관성이 보장된다. 검증 파이프라인의 순서가 명확하여 실수할 여지가 줄어든다.
- **부정적**: 기존에 MaxLength를 먼저 검사하던 코드를 수정해야 한다. 정규화 단계가 추가되어 파이프라인이 한 단계 길어진다.

### 확인

- `"  Hello  "`(공백 포함)이 MaxLength(5)를 통과하고 Trim된 `"Hello"`로 저장되는지 확인한다.
- `"  Hello World  "`(공백 제거 후 11자)가 MaxLength(5)에서 올바르게 거부되는지 확인한다.
- `CreateFromValidated`에 Trim되지 않은 문자열을 전달하면 거부되는지 확인한다.

## 옵션별 장단점

### 옵션 1: MaxLength 후 Normalize

- **장점**: 기존 코드 변경이 없다. 원본 입력의 길이를 먼저 검증하므로 과도하게 긴 입력을 조기에 차단할 수 있다.
- **단점**: 공백 포함 길이로 판단하여 유효한 입력을 잘못 거부한다. 정규화 후 실제 유효한 데이터가 검증 단계에서 탈락하는 오판(false rejection)이 발생한다.

### 옵션 2: Normalize 후 MaxLength

- **장점**: 정규화된 실제 데이터 기준으로 길이를 검증하므로 오판이 없다. DB 저장값과 검증 기준이 일치한다. 사용자 경험이 향상된다(불필요한 에러 메시지 감소).
- **단점**: 극단적으로 긴 입력(예: 공백 100만 자)이 정규화 단계까지 진행될 수 있다(별도의 원본 길이 상한 검사로 해결 가능).

### 옵션 3: Normalize 미적용

- **장점**: 정규화 로직이 불필요하여 파이프라인이 단순하다.
- **단점**: DB에 공백이 포함된 데이터가 저장되어 검색, 비교, Unique 제약 등에서 불일치가 발생한다. `"Hello"`와 `"  Hello  "`가 다른 값으로 취급되어 중복 데이터가 생긴다.

### 옵션 4: 단일 메서드에서 정규화와 검증 동시 수행

- **장점**: 순서 실수가 원천적으로 불가능하다. 호출 코드가 간결하다.
- **단점**: 정규화 전략(Trim, ToLower, 특수문자 제거 등)을 유연하게 조합할 수 없다. 검증 규칙과 정규화 로직이 결합되어 단일 책임 원칙에 위배된다. 파이프라인의 조합 가능성이 사라진다.
