# 도메인 용어 사전 작성 가이드

이 문서는 `DOMAIN-GLOSSARY.template.md` 템플릿을 사용하여 호스트 프로젝트의 도메인 용어 사전을 작성할 때 참고하는 가이드이다.

**대상 독자**: 새 호스트 프로젝트의 도메인 계층을 설계하는 개발자

## 사용 절차

1. **복사**: `DOMAIN-GLOSSARY.template.md`를 호스트 프로젝트의 도메인 루트에 `DOMAIN-GLOSSARY.md`로 복사한다.
2. **작성**: 이 가이드의 섹션별 지침을 참고하여 `{placeholder}`를 실제 도메인 내용으로 채운다.
3. **검토**: 하단의 [체크리스트](#체크리스트)로 자가 검증한다.
4. **정리**: 템플릿의 HTML 주석(`<!-- ... -->`)을 모두 제거한다.

## 섹션별 작성 가이드

### 섹션 1. 도메인 개요

**목적**: 도메인의 범위와 핵심 영역을 한 문장으로 요약한다.

**작성 기준**:
- 핵심 애그리거트를 **굵게** 나열하고 개수를 명시한다.
- 시스템의 성격을 한 단어로 표현한다 (예: 전자상거래 시스템, 예약 시스템).

**실제 예시** (01-SingleHost):

> 이 도메인은 **고객(Customer)**, **상품(Product)**, **주문(Order)**, **재고(Inventory)** 4개 핵심 영역으로 구성된 전자상거래 시스템이다.

**나쁜 예**:
- "이 시스템은 다양한 기능을 제공합니다" — 핵심 영역이 드러나지 않는다.
- 3문장 이상으로 장황하게 작성 — 개요는 1~2문장이면 충분하다.

---

### 섹션 2. 도메인 용어 사전 (Ubiquitous Language)

**목적**: 도메인에서 사용하는 모든 용어를 표로 정의하여 팀 내 공통 언어를 확립한다.

**작성 기준**:
- **유형**은 다음 중 하나: `애그리거트 루트`, `엔티티`, `값 객체`, `공유 엔티티`, `공유 값 객체`
- 영문 이름은 코드에서 사용할 클래스/레코드 이름과 일치시킨다.
- 정의에 핵심 제약을 간략히 포함한다 (최대 길이, 형식 등).
- 애그리거트 루트 → 하위 값 객체 순서로 그룹핑하여 읽기 쉽게 배치한다.

**실제 예시** (01-SingleHost, 일부 발췌):

| 용어 | 영문 | 유형 | 정의 |
|------|------|------|------|
| 고객 | Customer | 애그리거트 루트 | 주문을 생성할 수 있는 주체. 이름, 이메일, 신용한도를 가진다 |
| 고객명 | CustomerName | 값 객체 | 고객의 이름 (최대 100자, 공백 불가) |
| 금액 | Money | 공유 값 객체 | 양수인 화폐 금액. 덧셈/`Subtract`(`Fin<Money>` 반환)/`Multiply(decimal)` 연산 가능 |

**나쁜 예**:
- 유형 없이 용어만 나열 — 해당 용어가 엔티티인지 값 객체인지 알 수 없다.
- 정의에 구현 세부사항(DB 컬럼명, ORM 설정 등)을 포함 — 도메인 관점의 정의만 작성한다.

### 섹션 3. 애그리거트별 상세

**목적**: 각 애그리거트의 속성, 기능, 불변 조건, 조회 규칙을 상세하게 기술한다.

**작성 기준**:
- 애그리거트마다 기본 구조(`속성`, `기능`, `불변 조건`, `조회 규칙`)를 반복한다.
- **속성**은 섹션 2의 값 객체 용어를 사용한다.
- **기능**은 동사형으로 기술한다 (생성, 수정, 차감 등).
- **불변 조건**은 위반 시 어떻게 처리되는지 명시한다.
- **조회 규칙**은 Specification 이름을 괄호로 병기한다.
- 해당하는 경우 아래 **선택 항목**을 추가한다:
  - **자식 엔티티**: Aggregate 경계 내 Child Entity가 있을 때 속성/불변 조건을 기술한다. (예: Order → OrderLine)
  - **교차 규칙**: 다른 애그리거트와의 참조 방식(ID 참조 등)을 명시한다. (예: Order가 CustomerId, ProductId를 ID로만 참조)
  - **동시성**: 낙관적 잠금 등 동시성 제어가 있을 때 기술한다. (예: Inventory의 RowVersion)
  - **생명주기 관리**: 독립 Repository로 관리되거나 특수한 생명주기가 있을 때 기술한다. (예: Tag가 ITagRepository로 관리)
  - **설계 결정 근거**: 애그리거트 경계 설정, 승격/분리 등 설계 결정의 이유를 기록한다. (예: Tag를 독립 Aggregate Root로 승격한 이유)

**실제 예시** (01-SingleHost, 재고):

> ### 재고(Inventory)
>
> - **속성**: 상품ID, 보유수량(StockQuantity), 버전(RowVersion)
> - **기능**: 재고 생성, 재고 차감(`DeductStock` → `Fin<Unit>`), 재고 추가(`AddStock` → `Inventory` 반환, fluent)
> - **불변 조건**:
>   - 보유수량은 0 이상의 정수
>   - **재고 차감 시 보유수량보다 많은 수량을 차감할 수 없음** (부족 시 "InsufficientStock" 오류 — `Fin<Unit>`로 반환)
>   - 모든 수량 변경 시 수정 시각 갱신
> - **동시성**: 낙관적 잠금(RowVersion)으로 동시 재고 변경 충돌 감지
> - **조회 규칙**: 임계값 미만 재고 탐지 (InventoryLowStockSpec) — `StockQuantity < threshold` (미만, 이하 아님)

**실제 예시 — 선택 항목 사용** (01-SingleHost, 주문):

> ### 주문(Order)
>
> - **속성**: 고객ID(CustomerId), 주문라인 목록(OrderLines), 총액(TotalAmount), 배송주소, 주문상태(OrderStatus)
> - **기능**: 주문 생성(`Create` → `Fin<Order>`), 주문 확인(`Confirm` → `Fin<Unit>`), 주문 배송(`Ship` → `Fin<Unit>`), 주문 배달완료(`Deliver` → `Fin<Unit>`), 주문 취소(`Cancel` → `Fin<Unit>`)
> - **자식 엔티티 — 주문라인(OrderLine)**:
>   - **속성**: 상품ID(ProductId), 수량(Quantity), 단가(UnitPrice), 소계(LineTotal)
>   - **불변 조건**: 주문라인 수량은 반드시 0보다 커야 한다
>   - **소계 자동 계산**: LineTotal = UnitPrice × Quantity
> - **불변 조건**:
>   - 주문라인은 최소 1개 이상 필요
>   - **총액 = 모든 주문라인의 LineTotal 합계** (자동 계산, 직접 지정 불가)
>   - 배송주소는 비어있을 수 없으며 최대 500자
>   - **상태 전이 규칙**: Pending → Confirmed → Shipped → Delivered, Pending/Confirmed → Cancelled (허용되지 않는 전이 시 "InvalidOrderStatusTransition" 오류 — `Fin<Unit>`로 반환)
> - **교차 규칙**: 주문은 고객(CustomerId)을 ID로만 참조. 주문라인은 상품(ProductId)을 ID로만 참조 (다른 애그리거트를 직접 포함하지 않음)

**실제 예시 — 선택 항목 사용** (01-SingleHost, 태그):

> ### 태그(Tag)
>
> - **속성**: 태그명(TagName)
> - **기능**: 태그 생성만 지원
> - **생명주기 관리**: 독립 Aggregate Root로서 ITagRepository를 통해 생성/조회/수정/삭제 가능. Product는 TagId만 참조하며 Tag 객체를 직접 포함하지 않는다.
> - **설계 결정 근거**: Tag는 여러 Product에서 공유되므로 Product의 하위 Entity가 아닌 독립 Aggregate Root로 승격했다. 이를 통해 Tag의 독립적 생명주기(생성, 삭제)와 다대다 관계를 명확히 표현한다.

**나쁜 예**:
- 불변 조건 없이 속성만 나열 — 비즈니스 규칙이 누락된다.
- 반환 타입 미기재 — 실패 가능한 기능은 `Fin<T>` 등 반환 타입을 명시한다.

### 섹션 4. 애그리거트 간 관계

**목적**: 애그리거트 간의 참조/의존 관계를 시각적으로 표현한다.

**작성 기준**:
- ASCII 다이어그램으로 전체 관계를 한눈에 보여준다.
- 다이어그램 첫 줄에 범례(2줄)를 추가하여 표기법을 명시한다.
- **노드 표기법** — 4가지 유형을 `[]`(자식/포함)과 `**`(공유) 기호의 조합으로 구분한다:
  - `이름` = 애그리거트 루트
  - `[이름]` = 자식 엔티티 (Aggregate 경계 내 Child Entity)
  - `*이름*` = 공유 애그리거트 루트 (여러 Aggregate에서 ID로 참조)
  - `[*이름*]` = 공유 자식 엔티티 (여러 Aggregate에서 참조하는 Child Entity)
- **관계 표기법** — 2가지 화살표로 관계 성격을 구분한다:
  - `──▶` (실선) = ID 참조 또는 포함 관계
  - `┄┄▷` (점선, 수평) / `┆`·`▽` (점선, 수직) = 도메인 서비스 검증 (교차 Aggregate 비즈니스 규칙)
- **레이아웃 활용** — 수평·수직 배치를 의미 전달에 활용한다. 단순히 관계를 나열하는 것이 아니라, 배치 자체로 구조를 이해할 수 있게 구성한다:
  - **수평(──▶)**: 핵심 비즈니스 흐름을 왼쪽→오른쪽으로 배치한다. (예: Customer → Order → [OrderLine] → Product → *Tag*)
  - **수직(┆ ▽)**: 보조 관계나 도메인 서비스 검증처럼 핵심 흐름에서 분기되는 관계를 아래쪽으로 배치한다. (예: Order ┆▽ Inventory)
  - 수직 점선(`┆`/`▽`)은 수평 점선(`┄┄▷`)의 수직 대응이다. 방향이 필요하면 `▽`(아래), `△`(위)로 화살촉을 표현한다.
- 다이어그램 아래에 각 관계의 의미를 불릿으로 설명한다.

**실제 예시** (01-SingleHost):

```
범례: 이름 = 애그리거트 루트 | [이름] = 자식 엔티티 | *이름* = 공유 애그리거트 루트 | [*이름*] = 공유 자식 엔티티
범례: ──▶ = ID 참조/포함 | ┄┄▷ = 도메인 서비스 검증

Customer ──(고객ID 참조)──▶ Order ──(포함)──▶ [OrderLine] ◀──(상품ID 참조)── Product ──(태그ID 참조)──▶ *Tag*
                              ┆
                              ┆ (신용한도 검증)
                              ▽
                              Inventory ◀──(상품ID 참조)── Product
```

> - **고객 → 주문**: 주문은 고객ID를 참조. 고객의 신용한도 내에서만 주문 가능 (도메인 서비스로 검증)
> - **주문 → 주문라인**: Order가 OrderLine 컬렉션을 포함 (Aggregate 경계 내 Child Entity)
> - **주문라인 → 상품**: 주문라인은 상품 ID만 참조 (느슨한 결합)
> - **재고 → 상품**: 재고는 상품 ID만 참조. 상품과 분리된 이유는 변경 빈도가 높기 때문
> - **상품 → 태그**: 상품은 TagId만 참조. Tag는 독립 Aggregate Root로서 별도 Repository로 관리

### 섹션 5. 도메인 서비스

**목적**: 단일 애그리거트로 표현할 수 없는 교차 비즈니스 규칙을 기술한다.

**작성 기준**:
- 교차 애그리거트 규칙이 없으면 "해당 없음"으로 표기하거나 섹션을 제거한다.
- 서비스마다 `목적`, `규칙`, `반환 타입`을 기술한다.
- 왜 도메인 서비스여야 하는지(어떤 애그리거트에 걸쳐 있는지) 명시한다.
- 도메인 서비스가 있을 때, **Invariant 검증 위치 설계 결정** 표를 추가하여 단일 Aggregate 규칙 vs 교차 Aggregate 규칙의 검증 위치와 그 근거를 명시한다.

**실제 예시** (01-SingleHost):

> ### 주문 신용한도 검증 (OrderCreditCheckService)
>
> - **목적**: 고객의 주문 총액이 신용한도를 초과하지 않도록 보장
> - **규칙 1**: 단일 주문 금액이 고객 신용한도를 초과하면 거부 ("CreditLimitExceeded")
> - **규칙 2**: 기존 주문 총액 + 신규 주문 금액이 고객 신용한도를 초과하면 거부
> - **반환 타입**: `Fin<Unit>` (성공 시 `Unit`, 실패 시 `Error`)

**실제 예시 — Invariant 검증 위치 설계 결정** (01-SingleHost):

> | 검증 유형 | 위치 | 근거 |
> |-----------|------|------|
> | 단일 Aggregate 규칙 (DeductStock) | Inventory Aggregate 내부 | 재고 차감은 Inventory 자체 불변 조건이므로 Aggregate 내부에서 검증 |
> | 단일 Aggregate 규칙 (Order 상태 전이) | Order Aggregate 내부 | 상태 전이 가드는 Order 자체 불변 조건이므로 Aggregate 내부에서 검증 (DeductStock과 동일 패턴) |
> | 교차 Aggregate 규칙 (CreditCheck) | 도메인 서비스 (OrderCreditCheckService) | Customer의 신용한도와 Order의 금액을 함께 검증해야 하므로, 어느 한쪽 Aggregate에 넣으면 다른 Aggregate에 대한 의존성이 생김. 도메인 서비스로 분리하여 양쪽 Aggregate를 순수 함수적으로 검증 |

### 섹션 6. 도메인 이벤트

**목적**: 도메인 상태 변경 시 발행되는 이벤트를 정리한다.

**작성 기준**:
- 도메인 이벤트가 없으면 "해당 없음"으로 표기하거나 섹션을 제거한다.
- 이벤트명은 `{Aggregate}.{Event}` 형식으로 통일한다.
- 발생 시점과 포함 정보를 빠짐없이 기재한다.

**실제 예시** (01-SingleHost, 일부 발췌):

| 이벤트 | 발생 시점 | 포함 정보 |
|--------|-----------|-----------|
| Product.CreatedEvent | 상품 생성 | 상품ID, 상품명, 가격 |
| Product.UpdatedEvent | 상품 수정 | 상품ID, 상품명, 변경 전 가격, 변경 후 가격 |
| Inventory.StockDeductedEvent | 재고 차감 | 재고ID, 상품ID, 차감 수량 |

### 섹션 7. 핵심 제약조건(불변 조건) 요약

**목적**: 섹션 3에 흩어진 불변 조건을 한 장의 표로 모아 빠르게 조회할 수 있게 한다.

**작성 기준**:
- 섹션 3의 모든 불변 조건을 빠짐없이 포함한다.
- `분류`는 값 객체 또는 비즈니스 규칙 단위로 그룹핑한다.
- `오류`는 도메인에서 사용하는 오류 키 또는 처리 방식을 기재한다.

**실제 예시** (01-SingleHost, 일부 발췌):

| 분류 | 제약조건 | 오류 |
|------|----------|------|
| 금액 | 반드시 양수. `Subtract` 결과가 음수이면 `Fin<Money>` Fail 반환 | 검증 실패 |
| 주문라인 수량 | 주문라인의 수량은 반드시 0보다 커야 함 | InvalidQuantity |
| 주문라인 최소 개수 | 주문에는 최소 1개의 주문라인 필요 | EmptyOrderLines |
| 재고 부족 | 차감 수량 ≤ 현재 보유수량 | InsufficientStock |
| 태그 중복 | 동일 상품에 같은 태그 중복 부여 불가 (멱등) | 무시(멱등 처리) |
| 재고 동시성 | 낙관적 잠금(RowVersion)으로 충돌 감지 | 동시성 충돌 오류 |
