# 도메인 모델 요구사항

## 1. 개요

본 시스템은 **전자상거래 주문 처리**를 위한 단일 Bounded Context로 구성된다.
**고객**, **상품**, **주문**, **재고**, **태그** 5개 핵심 영역을 포함하며, 단일 배포 단위로 관리된다.

## 2. 도메인 용어 정의 (Ubiquitous Language)

| 용어 | 영문 | 유형 | 요구사항 정의 |
|------|------|------|---------------|
| 고객 | Customer | 애그리거트 루트 | 주문을 생성할 수 있는 주체. 이름, 이메일, 신용한도를 가진다 |
| 고객명 | CustomerName | 값 객체 | 고객의 이름 (최대 100자, 공백 불가) |
| 이메일 | Email | 값 객체 | 고객의 이메일 주소. 유효한 형식이어야 한다 (최대 320자) |
| 신용한도 | CreditLimit | 값 객체 | 고객이 주문할 수 있는 최대 총 금액 |
| 상품 | Product | 애그리거트 루트 | 판매 대상. 이름, 설명, 가격, 태그 목록을 가진다 |
| 상품명 | ProductName | 값 객체 | 상품의 이름 (최대 100자, 공백 불가) |
| 상품설명 | ProductDescription | 값 객체 | 상품의 상세 설명 (최대 1000자, 빈 값 허용) |
| 태그 | Tag | 애그리거트 루트 | 상품에 부여할 수 있는 분류 라벨. 독립 생명주기를 가진다 |
| 태그명 | TagName | 값 객체 | 태그의 이름 (최대 50자, 공백 불가) |
| 주문 | Order | 애그리거트 루트 | 고객이 상품을 구매하는 행위의 기록. 다중 주문라인을 포함한다 |
| 주문라인 | OrderLine | 엔티티 | 주문 내 개별 상품의 수량·단가·소계 |
| 주문상태 | OrderStatus | 값 객체 | 주문의 처리 상태 (Pending/Confirmed/Shipped/Delivered/Cancelled) |
| 배송주소 | ShippingAddress | 값 객체 | 주문의 배송 목적지 (최대 500자, 공백 불가) |
| 재고 | Inventory | 애그리거트 루트 | 특정 상품의 현재 보유 수량. 동시성 제어를 포함한다 |
| 금액 | Money | 공유 값 객체 | 양수인 화폐 금액. 덧셈, 뺄셈, 곱셈, 합산 연산을 지원해야 한다 |
| 수량 | Quantity | 공유 값 객체 | 0 이상의 정수. 덧셈/뺄셈 연산을 지원해야 한다 |

## 3. 애그리거트별 요구사항

### 3.1 고객 (Customer)

#### 데이터 요구사항

- **CUS-R01**: 고객은 고객명, 이메일, 신용한도를 필수 속성으로 가져야 한다.

#### 기능 요구사항

- **CUS-R02**: 고객을 생성할 수 있어야 한다.
- **CUS-R03**: 신용한도를 변경할 수 있어야 하며, 수정 시각을 기록해야 한다.
- **CUS-R04**: 이메일을 변경할 수 있어야 하며, 수정 시각을 기록해야 한다.

#### 비즈니스 규칙

- **CUS-R05**: 고객명은 비어있을 수 없으며 최대 100자여야 한다.
- **CUS-R06**: 이메일은 유효한 형식이어야 하며 최대 320자여야 한다.
- **CUS-R07**: 이메일은 소문자로 정규화하여 저장해야 한다.
- **CUS-R08**: 신용한도는 반드시 양수여야 한다.
- **CUS-R09**: 동일 이메일을 가진 고객이 중복 존재할 수 없다.

#### 조회 요구사항

- **CUS-R10**: 이메일로 고객 존재 여부를 확인할 수 있어야 한다.

### 3.2 상품 (Product)

#### 데이터 요구사항

- **PRD-R01**: 상품은 상품명, 상품설명, 가격, 태그 목록을 속성으로 가져야 한다.

#### 기능 요구사항

- **PRD-R02**: 상품을 생성할 수 있어야 한다.
- **PRD-R03**: 상품의 이름, 설명, 가격을 수정할 수 있어야 하며, 수정 시각을 기록해야 한다.
- **PRD-R04**: 상품에 태그를 할당할 수 있어야 한다.
- **PRD-R05**: 상품에서 태그를 해제할 수 있어야 한다 (존재하지 않는 태그 해제 시 무시, 멱등 처리).

#### 비즈니스 규칙

- **PRD-R06**: 상품명은 비어있을 수 없으며 최대 100자여야 한다.
- **PRD-R07**: 상품설명은 최대 1000자여야 한다 (빈 값 허용).
- **PRD-R08**: 가격은 반드시 양수여야 한다.
- **PRD-R09**: 동일 태그를 중복 부여할 수 없다 (멱등 처리).
- **PRD-R10**: 동일 상품명을 가진 상품이 중복 존재할 수 없다.

#### 조회 요구사항

- **PRD-R11**: 상품명으로 정확 일치 검색할 수 있어야 한다.
- **PRD-R12**: 상품명 고유성을 검증할 수 있어야 한다 (수정 시 자기 자신 제외).
- **PRD-R13**: 가격 범위로 검색할 수 있어야 한다 (최소가격 ≤ 가격 ≤ 최대가격).

### 3.3 주문 (Order)

#### 데이터 요구사항

- **ORD-R01**: 주문은 고객 ID, 주문라인 목록, 총액, 배송주소, 주문상태를 속성으로 가져야 한다.

#### 기능 요구사항

- **ORD-R02**: 주문을 생성할 수 있어야 한다 (실패 시 오류를 반환해야 한다).
- **ORD-R03**: 주문을 확인할 수 있어야 한다 (실패 시 오류를 반환해야 한다).
- **ORD-R04**: 주문을 배송 처리할 수 있어야 한다 (실패 시 오류를 반환해야 한다).
- **ORD-R05**: 주문 배달 완료를 기록할 수 있어야 한다 (실패 시 오류를 반환해야 한다).
- **ORD-R06**: 주문을 취소할 수 있어야 한다 (실패 시 오류를 반환해야 한다).

#### 자식 엔티티 — 주문라인 (OrderLine)

- **ORD-R07**: 주문라인은 상품 ID, 수량, 단가, 소계를 속성으로 가져야 한다.
- **ORD-R08**: 주문라인의 수량은 반드시 0보다 커야 한다.
- **ORD-R09**: 주문라인의 소계는 단가 × 수량으로 자동 계산되어야 한다.

#### 비즈니스 규칙

- **ORD-R10**: 주문라인은 최소 1개 이상 포함해야 한다.
- **ORD-R11**: 총액은 모든 주문라인의 소계 합계로 자동 계산되어야 한다 (직접 지정 불가).
- **ORD-R12**: 배송주소는 비어있을 수 없으며 최대 500자여야 한다.
- **ORD-R13**: 주문 상태 전이는 Pending → Confirmed → Shipped → Delivered, Pending → Cancelled, Confirmed → Cancelled만 허용해야 한다.
- **ORD-R14**: 허용되지 않는 상태 전이 시 오류를 반환해야 한다.
- **ORD-R15**: 주문은 고객을 ID로만 참조해야 한다.
- **ORD-R16**: 주문라인은 상품을 ID로만 참조해야 한다.
- **ORD-R17**: 주문 생성 후 주문라인을 변경할 수 없다 (생성 시점에 확정).

### 3.4 재고 (Inventory)

#### 데이터 요구사항

- **INV-R01**: 재고는 상품 ID, 보유수량, 버전을 속성으로 가져야 한다.

#### 기능 요구사항

- **INV-R02**: 재고를 생성할 수 있어야 한다.
- **INV-R03**: 재고를 차감할 수 있어야 한다 (실패 시 오류를 반환해야 한다).
- **INV-R04**: 재고를 추가할 수 있어야 한다.

#### 비즈니스 규칙

- **INV-R05**: 보유수량은 0 이상의 정수여야 한다.
- **INV-R06**: 재고 차감 시 보유수량보다 많은 수량을 차감할 수 없다 (부족 시 오류 반환).
- **INV-R07**: 모든 수량 변경 시 수정 시각을 기록해야 한다.
- **INV-R08**: 낙관적 잠금으로 동시 재고 변경 충돌을 감지해야 한다.

#### 조회 요구사항

- **INV-R09**: 임계값 미만의 재고를 탐지할 수 있어야 한다 (보유수량 < 임계값).

### 3.5 태그 (Tag)

#### 데이터 요구사항

- **TAG-R01**: 태그는 태그명을 필수 속성으로 가져야 한다.
- **TAG-R02**: 태그는 생성 시각과 수정 시각을 기록해야 한다.

#### 기능 요구사항

- **TAG-R03**: 태그를 생성할 수 있어야 한다.
- **TAG-R04**: 태그 이름을 변경할 수 있어야 하며, 수정 시각을 기록해야 한다.
- **TAG-R05**: 태그를 조회, 삭제할 수 있어야 한다.

#### 비즈니스 규칙

- **TAG-R06**: 태그는 독립 생명주기를 가지며, Product에서 TagId로 참조한다.
- **TAG-R07**: 상품은 태그 ID만 참조하며 태그 객체를 직접 포함하지 않는다.

## 4. 애그리거트 간 관계

```
고객 ──(고객ID 참조)──▶ 주문 ──(포함)──▶ [주문라인] ◀──(상품ID 참조)── 상품 ──(태그ID 참조)──▶ 태그
                              ┆
                              ┆ (신용한도 검증)
                              ▽
                              재고 ◀──(상품ID 참조)── 상품
```

- **REL-R01**: 주문은 고객 ID를 참조해야 한다.
- **REL-R02**: 고객의 신용한도 내에서만 주문이 가능해야 한다.
- **REL-R03**: 주문은 주문라인 컬렉션을 포함해야 한다.
- **REL-R04**: 주문라인은 상품 ID만 참조해야 한다 (느슨한 결합).
- **REL-R05**: 재고는 상품 ID만 참조해야 한다.
- **REL-R06**: 상품은 태그 ID만 참조해야 한다.
- **REL-R07**: 태그는 독립적으로 관리되어야 한다.

## 5. 교차 도메인 비즈니스 규칙

### 주문 신용한도 검증

- **CRS-R01**: 고객의 주문 총액이 신용한도를 초과하지 않도록 보장해야 한다.
- **CRS-R02**: 단일 주문 금액이 고객 신용한도를 초과하면 거부해야 한다.
- **CRS-R03**: 고객의 기존 주문 총액과 신규 주문 금액의 합이 고객 신용한도를 초과하면 거부해야 한다.
- **CRS-R04**: 신용한도 검증은 고객과 주문 두 영역에 걸쳐 있으므로 도메인 서비스로 구현해야 한다.
- **CRS-R05**: 검증 실패 시 오류를 반환해야 한다.

### 검증 위치 원칙

| 검증 유형 | 위치 | 근거 |
|-----------|------|------|
| 단일 애그리거트 규칙 (재고 차감) | 재고 애그리거트 내부 | 자체 불변 조건이므로 내부에서 검증 |
| 단일 애그리거트 규칙 (주문 상태 전이) | 주문 애그리거트 내부 | 자체 불변 조건이므로 내부에서 검증 |
| 교차 애그리거트 규칙 (신용한도 검증) | 도메인 서비스 | 두 애그리거트의 데이터를 함께 검증해야 하므로 서비스로 분리 |

## 6. 도메인 이벤트 요구사항

다음 상태 변경 시 해당 이벤트를 발행해야 한다.

- **EVT-R01**: 고객 생성 시 고객 생성 이벤트를 발행해야 한다 (고객 ID, 고객명, 이메일 포함).
- **EVT-R02**: 신용한도 변경 시 신용한도 변경 이벤트를 발행해야 한다 (고객 ID, 이전 신용한도, 새 신용한도 포함).
- **EVT-R03**: 이메일 변경 시 이메일 변경 이벤트를 발행해야 한다 (고객 ID, 이전 이메일, 새 이메일 포함).
- **EVT-R04**: 상품 생성 시 상품 생성 이벤트를 발행해야 한다 (상품 ID, 상품명, 가격 포함).
- **EVT-R05**: 상품 수정 시 상품 수정 이벤트를 발행해야 한다 (상품 ID, 상품명, 변경 전 가격, 변경 후 가격 포함).
- **EVT-R06**: 태그 할당 시 태그 할당 이벤트를 발행해야 한다 (상품 ID, 태그 ID 포함).
- **EVT-R07**: 태그 해제 시 태그 해제 이벤트를 발행해야 한다 (상품 ID, 태그 ID 포함).
- **EVT-R08**: 태그 생성 시 태그 생성 이벤트를 발행해야 한다 (태그 ID, 태그명 포함).
- **EVT-R09**: 태그 이름 변경 시 태그 이름 변경 이벤트를 발행해야 한다 (태그 ID, 이전 태그명, 새 태그명 포함).
- **EVT-R10**: 주문 생성 시 주문 생성 이벤트를 발행해야 한다 (주문 ID, 고객 ID, 주문라인 목록, 총액 포함).
- **EVT-R11**: 주문 확인 시 주문 확인 이벤트를 발행해야 한다 (주문 ID 포함).
- **EVT-R12**: 주문 배송 시 주문 배송 이벤트를 발행해야 한다 (주문 ID 포함).
- **EVT-R13**: 주문 배달 완료 시 주문 배달 완료 이벤트를 발행해야 한다 (주문 ID 포함).
- **EVT-R14**: 주문 취소 시 주문 취소 이벤트를 발행해야 한다 (주문 ID 포함).
- **EVT-R15**: 재고 생성 시 재고 생성 이벤트를 발행해야 한다 (재고 ID, 상품 ID, 수량 포함).
- **EVT-R16**: 재고 차감 시 재고 차감 이벤트를 발행해야 한다 (재고 ID, 상품 ID, 차감 수량 포함).
- **EVT-R17**: 재고 추가 시 재고 추가 이벤트를 발행해야 한다 (재고 ID, 상품 ID, 추가 수량 포함).

## 7. 제약조건 요약

| 분류 | 제약조건 | 관련 요구사항 |
|------|----------|---------------|
| 금액 | 반드시 양수여야 한다. 뺄셈 결과가 음수이면 오류를 반환해야 한다 | CUS-R08, PRD-R08 |
| 수량 | 0 이상의 정수여야 한다. 뺄셈 시 0 미만이 되지 않도록 0으로 클램핑해야 한다 | INV-R05 |
| 주문라인 수량 | 주문라인의 수량은 반드시 0보다 커야 한다 | ORD-R08 |
| 주문라인 최소 개수 | 주문에는 최소 1개의 주문라인이 포함되어야 한다 | ORD-R10 |
| 이메일 형식 | 유효한 이메일 형식이어야 하며, 소문자로 정규화해야 한다 | CUS-R06, CUS-R07 |
| 주문 총액 | 모든 주문라인 소계의 합계로 자동 계산되어야 한다 | ORD-R11 |
| 신용한도 초과 | 고객의 기존 주문 총액 + 신규 주문 금액이 신용한도를 초과할 수 없다 | CRS-R03 |
| 재고 부족 | 차감 수량이 현재 보유수량을 초과할 수 없다 | INV-R06 |
