## 개요

### 목적

가이드 문서는 사용자가 프로젝트의 도구, 명령어, 규칙을 쉽게 이해하고 활용할 수 있도록 돕습니다.

### 대상 독자

- 프로젝트를 처음 접하는 개발자
- 특정 도구나 명령어의 사용법을 찾는 개발자
- 팀의 작업 규칙을 확인하려는 개발자

### 문서 위치

```
Docs/Guides/
├── Guide-Writing-Guide.md      # 이 문서 (메타 가이드)
├── Git-Commands.md             # Git 명령어 가이드
├── Git-Commit.md               # Git 커밋 가이드
└── ...
```

### 다른 가이드와의 차이

| 항목 | `.claude/guides/` | `Docs/Guides/` (이 가이드) |
|------|-------------------|---------------------------|
| 대상 | 개발자, AI 어시스턴트 | 사용자, 개발자 |
| 목적 | 아키텍처, 패턴 문서화 | 사용법, 명령어 가이드 |
| 파일명 | `guide-{주제}.md` | `{주제}-{세부주제}.md` |
| 섹션 구분 | `---` | `<br/>` |
| 스타일 | 기술 문서 | 사용자 매뉴얼 |

## 요약

### 주요 형식

**파일명:**
```
{주제}-{세부주제}.md

예시: Git-Commands.md, Build-Globaljson.md
```

**섹션 구분:**
```markdown
<br/>  # 각 ## 섹션 끝에 추가
```

**요약 구조:**
```markdown
### 주요 명령
{명령어 예시}

### 주요 절차
{단계별 워크플로우}

### 주요 개념
{핵심 개념}
```

### 주요 절차

**1. 새 가이드 작성:**
```bash
# 1. 파일 생성
# Docs/Guides/{주제}-{세부주제}.md

# 2. 기본 구조 작성
# - 제목 및 한 줄 설명
# - 목차
# - 개요
# - 요약 (주요 명령 + 주요 절차 + 주요 개념)
# - 트러블슈팅
# - FAQ

# 3. Git 커밋
git add Docs/Guides/{파일명}.md
git commit -m "docs: {주제} 가이드 추가"
```

**2. 기존 가이드 수정:**
```bash
# 1. 파일 수정

# 2. 목차 업데이트 (섹션 추가/삭제 시)

# 3. <br/> 구분 유지 확인

# 4. Git 커밋
git add Docs/Guides/{파일명}.md
git commit -m "docs: {주제} 가이드 업데이트"
```

### 주요 개념

**1. 요약 섹션 구성 (3개 섹션)**
- **주요 명령**: 사용자가 바로 실행할 수 있는 명령어 (빠른 참조)
- **주요 절차**: 처음부터 끝까지 단계별 워크플로우 (가이드)
- **주요 개념**: 이해해야 할 핵심 원칙과 이론

**2. 형식 규칙**

| 항목 | `.claude/guides/` | `Docs/Guides/` |
|------|-------------------|----------------|
| 섹션 구분 | `---` | `<br/>` |
| 파일명 | `guide-{주제}.md` | `{주제}-{세부주제}.md` |
| 요약 형식 | 자유 형식 | 주요 명령 + 주요 절차 + 주요 개념 |

**3. 필수 섹션**
- 제목과 한 줄 설명
- 목차 (모든 `##` 섹션)
- 개요 (선택)
- 요약 (필수: 3개 섹션)
- 트러블슈팅 (필수)
- FAQ (필수: 3~10개)

**4. 템플릿 선택**

| 가이드 유형 | 요약 중점 |
|------------|----------|
| 명령어/도구 | 명령어 예시 + 단계별 절차 |
| 규칙/프로세스 | 절차 + 개념 설명 |

## 파일 명명 규칙

### 형식

```
{주제}-{세부주제}.md
```

### 규칙

| 항목 | 규칙 | 예시 |
|------|------|------|
| 주제 | 대문자로 시작 | `Git`, `Docker`, `API` |
| 세부주제 | 대문자로 시작 | `Commands`, `Commit`, `Setup` |
| 구분자 | 하이픈 (`-`) 사용 | `Git-Commands.md` |
| 확장자 | `.md` (Markdown) | `Git-Commit.md` |

### 예시

```
Git-Commands.md           # Git 명령어 가이드
Git-Commit.md             # Git 커밋 가이드
Docker-Setup.md           # Docker 설정 가이드
API-Reference.md          # API 참조 가이드
Testing-Guidelines.md     # 테스트 가이드라인
Code-Review.md            # 코드 리뷰 가이드
```

## 문서 구조

### 표준 구조

```markdown
# {제목}

{한 줄 설명 - 이 문서가 무엇을 설명하는지}

## 목차
- [섹션1](#섹션1)
- [섹션2](#섹션2)
...
- [트러블슈팅](#트러블슈팅)
- [FAQ](#faq)

<br/>

## 섹션1

{내용}

<br/>

## 섹션2

{내용}

<br/>

## 트러블슈팅

{문제 해결}

<br/>

## FAQ

{자주 묻는 질문}
```

### 필수 섹션

모든 가이드 문서는 다음 섹션을 포함해야 합니다:

| 섹션 | 위치 | 설명 |
|------|------|------|
| 제목 | 최상단 | `# {제목}` 형식 |
| 한 줄 설명 | 제목 바로 아래 | 문서의 목적을 한 문장으로 설명 |
| 목차 | 설명 다음 | 모든 `##` 레벨 섹션 링크 |
| 트러블슈팅 | 끝에서 2번째 | 흔한 문제와 해결 방법 |
| FAQ | 마지막 | 자주 묻는 질문 (3~10개) |

### 선택 섹션

문서 성격에 따라 포함할 수 있습니다:

| 섹션 | 사용 시기 | 예시 |
|------|----------|------|
| 개요 | 배경 설명이 필요할 때 | 도구 소개, 개념 설명 |
| 기본 명령어 | 명령어 가이드 | Git, Docker 등 |
| 사용 예시 | 실습이 필요할 때 | 코드 샘플, 실행 예제 |
| 베스트 프랙티스 | 권장 사항 제공 | 코딩 규칙, 리뷰 가이드 |

## 필수 섹션

### 1. 제목과 한 줄 설명

```markdown
# Git 명령어 가이드

이 문서는 Git의 주요 명령어와 사용법을 설명합니다.
```

**규칙:**
- 제목: `# {주제} {세부주제}` 형식
- 한 줄 설명: 제목 바로 아래, 빈 줄로 구분
- 간결하고 명확하게 작성

### 2. 목차

```markdown
## 목차
- [개요](#개요)
- [기본 명령어](#기본-명령어)
- [고급 명령어](#고급-명령어)
- [트러블슈팅](#트러블슈팅)
- [FAQ](#faq)
```

**규칙:**
- 모든 `##` 레벨 섹션을 포함
- 앵커는 소문자, 공백은 하이픈으로 변환
- 순서는 문서 구조를 따름

### 3. 트러블슈팅

```markdown
## 트러블슈팅

### 커밋 메시지를 잘못 작성했을 때

아직 push하지 않은 경우:
```bash
git commit --amend -m "올바른 메시지"
```

이미 push한 경우:
```bash
git commit --amend -m "올바른 메시지"
git push --force-with-lease  # 주의: 팀원과 협의 필요
```

> **주의**: force push는 공유 브랜치에서 사용하지 마세요.
```

**규칙:**
- 문제 상황을 `###` 제목으로 명시
- 해결 방법을 코드 블록과 함께 제공
- 주의사항은 인용문(`>`)으로 강조

### 4. FAQ

```markdown
## FAQ

### Q1. 명령어 A와 명령어 B의 차이점은 무엇인가요?

| 명령어 | 동작 |
|--------|------|
| 명령어 A | 설명 A |
| 명령어 B | 설명 B |

### Q2. 특정 상황에서 어떻게 해야 하나요?

1. 첫 번째 단계
2. 두 번째 단계

```bash
# 예시 코드
```
```

**규칙:**
- 질문: `### Q{번호}. {질문}?` 형식
- 답변: 질문 바로 아래, 표/리스트/코드 블록 활용
- 3~10개 질문 권장

## 형식 규칙

### 섹션 구분

각 `##` 레벨 섹션 끝에 `<br/>` 태그를 추가합니다.

```markdown
## 섹션 1

내용...

<br/>

## 섹션 2

내용...

<br/>
```

**중요**: `.claude/guides/`는 `---`를 사용하지만, `Docs/Guides/`는 `<br/>`를 사용합니다.

### 표 (Table)

명령어, 옵션, 비교 정보에 사용합니다.

```markdown
| 명령어 | 설명 | 예시 |
|--------|------|------|
| `git status` | 작업 상태 확인 | `git status` |
| `git add` | 파일 스테이징 | `git add .` |
```

**규칙:**
- 헤더와 데이터를 명확히 구분
- 코드는 백틱(`)으로 감싸기
- 좌측 정렬 사용

### 리스트

```markdown
### 일반 리스트
- 항목 1
- 항목 2
  - 하위 항목 2-1
  - 하위 항목 2-2

### 번호 리스트 (순서가 중요할 때)
1. 첫 번째 단계
2. 두 번째 단계
3. 세 번째 단계

### 체크리스트 (요약 섹션용)
- [ ] 완료 항목 1
- [ ] 완료 항목 2
```

### 강조

```markdown
**굵게** - 중요한 키워드, 강조
*기울임* - 부연 설명
`코드` - 명령어, 파일명, 변수명
```

**예시:**
- **중요**: 이 명령어는 데이터를 삭제합니다.
- `git status` 명령어를 실행하세요.
- *참고*: 이 옵션은 선택사항입니다.

### 인용

중요한 주의사항이나 팁을 강조할 때 사용합니다.

```markdown
> **주의**: 이 명령어는 되돌릴 수 없습니다.
> **팁**: 이 방법이 더 효율적입니다.
```

## 코드 블록 작성

### 언어 명시

항상 코드 블록에 언어를 명시합니다.

````markdown
```bash
# Shell 명령어
git status
```

```javascript
// JavaScript 코드
const result = doSomething();
```

```json
{
  "key": "value"
}
```
````

### 주석 스타일

사용자가 이해하기 쉽도록 주석을 추가합니다.

```bash
# 1. 변경사항 확인
git status

# 2. 파일 스테이징
git add .

# 3. 커밋
git commit -m "feat: 새 기능 추가"
```

### 예시 구분

좋은 예시와 나쁜 예시를 명확히 구분합니다.

```markdown
**좋은 예시**:
```bash
git commit -m "feat: 로그인 기능 추가"
```

**나쁜 예시**:
```bash
git commit -m "업데이트"
```
```

또는 체크마크/X 표시 사용:

```markdown
```bash
# ✓ 좋은 예시
git commit -m "feat: 로그인 기능 추가"

# ✗ 나쁜 예시
git commit -m "업데이트"
```
```

## FAQ

### Q1. git status는 무엇을 보여주나요?

`git status`는 현재 작업 디렉토리의 상태를 보여줍니다:
- 스테이징된 파일
- 수정된 파일
- 추적되지 않는 파일
```

### 나쁜 예시

```markdown
# Git

git을 사용하는 방법

git status
git add
git commit

끝
```

**문제점:**
- 목차 없음
- `<br/>` 구분 없음
- 설명 부족
- 코드 블록 없음
- 트러블슈팅/FAQ 없음

## 트러블슈팅

### 목차 링크가 작동하지 않을 때

**원인**: 앵커가 잘못되었습니다.

**해결**:
- 섹션 제목을 소문자로 변환
- 공백을 하이픈(`-`)으로 변경
- 특수문자 제거

```markdown
## 기본 명령어  → #기본-명령어
## FAQ         → #faq
## Git 설치    → #git-설치
```

### 코드 블록이 제대로 표시되지 않을 때

**원인**: 언어 명시를 빠뜨렸거나 백틱이 부족합니다.

**해결**:

````markdown
```bash
# 올바른 예시
git status
```
````

### 표가 깨져 보일 때

**원인**: 열 구분자(`|`)가 잘못되었습니다.

**해결**:

```markdown
| 열1 | 열2 | 열3 |
|-----|-----|-----|
| 값1 | 값2 | 값3 |
```

**확인 사항:**
- 각 행의 열 개수가 동일한지 확인
- 헤더와 구분선이 있는지 확인

### 섹션 구분이 명확하지 않을 때

**원인**: `<br/>` 태그를 빠뜨렸습니다.

**해결**:

```markdown
## 섹션 1

내용...

<br/>  ← 추가

## 섹션 2

내용...

<br/>  ← 추가
```

## FAQ

### Q1. 새 가이드 문서는 어떻게 시작하나요?

**A:** 템플릿 섹션의 기본 템플릿 또는 명령어 가이드 템플릿을 복사하여 시작하세요. 주제에 맞게 섹션을 추가/제거합니다.

### Q2. 파일명은 어떻게 정하나요?

**A:** `{주제}-{세부주제}.md` 형식을 사용합니다:
- ✓ `Git-Commands.md`
- ✓ `Docker-Setup.md`
- ✗ `git_commands.md` (언더스코어 사용 금지)
- ✗ `gitcommands.md` (구분 없음)

### Q3. 섹션 구분은 `---`와 `<br/>` 중 무엇을 사용하나요?

**A:** `Docs/Guides/` 폴더에서는 **`<br/>`를 사용**합니다.

| 폴더 | 구분자 |
|------|--------|
| `.claude/guides/` | `---` |
| `Docs/Guides/` | `<br/>` |

### Q4. FAQ는 몇 개가 적당한가요?

**A:** 3~10개를 권장합니다:
- 최소 3개: 기본적인 질문
- 권장 5~7개: 다양한 질문
- 최대 10개: 상세한 가이드

### Q5. 코드 블록에는 항상 언어를 명시해야 하나요?

**A:** 네, 항상 명시하세요:

````markdown
```bash
# Shell 명령어
```

```javascript
// JavaScript 코드
```

```json
{...}
```
````

### Q6. 표와 리스트 중 어느 것을 사용해야 하나요?

**A:** 상황에 따라 선택하세요:

| 사용 | 형식 |
|------|------|
| 비교/매핑 정보 | 표 (Table) |
| 단순 나열 | 리스트 (List) |
| 순차적 절차 | 번호 리스트 |

### Q7. 기존 문서를 수정할 때 주의사항은?

**A:** 다음을 확인하세요:
- 목차 업데이트 (섹션 추가/삭제 시)
- `<br/>` 구분 유지
- 일관된 형식 유지
- 기존 스타일 따르기

### Q8. 트러블슈팅 섹션에는 무엇을 포함하나요?

**A:** 사용자가 겪을 수 있는 흔한 문제와 해결 방법:
- 명령어 실행 오류
- 설정 문제
- 예상치 못한 동작
- 권한 문제

각 문제마다 해결 방법을 코드 예시와 함께 제공하세요.

### Q9. 예시는 얼마나 상세해야 하나요?

**A:** 사용자가 복사하여 바로 사용할 수 있을 정도로 완전해야 합니다:

```bash
# 좋은 예시 - 완전한 명령어와 설명
git add src/main.js
git commit -m "feat: 로그인 기능 추가"
git push origin main

# 나쁜 예시 - 불완전
git add <파일>  # 구체적이지 않음
```

### Q10. 다른 가이드 문서를 참조하려면 어떻게 하나요?

**A:** 상대 경로로 링크하세요:

```markdown
[Git 명령어 가이드](./Git-Commands.md)를 참조하세요.

자세한 내용은 [Git 커밋 가이드](./Git-Commit.md#커밋-타입)의 커밋 타입 섹션을 확인하세요.
```
