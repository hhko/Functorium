## 요약

### 주요 형식

**파일명:**
```
{번호}-{주제}.md

예시: 15a-unit-testing.md, A01-vscode-debugging.md
```

**섹션 구분:**
```markdown
---  # 각 ## 섹션 사이에 추가
```

**요약 구조:**
```markdown
### 주요 명령
{명령어 예시}

### 주요 절차
{단계별 워크플로우}

### 주요 개념
{핵심 개념}
```

### 주요 절차

**1. 새 가이드 작성:**
```bash
# 1. 파일 생성 (Docs/guides/{번호}-{주제}.md)
# 2. 기본 구조 작성 (제목, 목차, 요약, 트러블슈팅, FAQ)
# 3. Git 커밋
```

**2. 기존 가이드 수정:**
```bash
# 1. 파일 수정
# 2. 목차 업데이트 (섹션 추가/삭제 시)
# 3. --- 구분 유지 확인
# 4. Git 커밋
```

### 주요 개념

**1. 요약 섹션 구성 (3개 섹션)**
- **주요 명령**: 사용자가 바로 실행할 수 있는 명령어 (빠른 참조)
- **주요 절차**: 처음부터 끝까지 단계별 워크플로우 (가이드)
- **주요 개념**: 이해해야 할 핵심 원칙과 이론

**2. 필수 섹션**
- 제목과 한 줄 설명
- 목차 (모든 `##` 섹션)
- 요약 (필수: 3개 섹션)
- 트러블슈팅 (필수)
- FAQ (필수: 3~10개)




## 파일 명명 규칙

### 형식

```
{번호}-{주제}.md
```

### 규칙

| 항목 | 규칙 | 예시 |
|------|------|------|
| 번호 | 두 자리 숫자 (카테고리별) | `00`, `15`, `24` |
| 주제 | 소문자, 하이픈으로 연결 | `unit-testing`, `vscode-debugging` |
| 확장자 | `.md` (Markdown) | `15a-unit-testing.md` |

### 번호 체계

| 범위 | 카테고리 |
|------|---------|
| `00` | 메타 가이드 (작성 규칙 등) |
| `01-03` | 프로젝트 구조, 솔루션, 도구 |
| `04-14` | 아키텍처, DDD, 패턴 |
| `15-16` | 테스트 |
| `17-22` | 관측성, DTO, 진단 |
| `A01-A99` | 부록 (학습 경로 외 참고 자료) |

### 예시

```
00-writing-guide.md            # 문서 작성 가이드
15a-unit-testing.md             # 단위 테스트 가이드
15b-integration-testing.md     # 통합 테스트 가이드 (하위 번호)
A01-vscode-debugging.md         # VSCode 디버깅 가이드
A02-git-reference.md            # Git 명령어 참조
```




## 문서 구조

### 표준 구조

```markdown
# {제목}

{한 줄 설명 - 이 문서가 무엇을 설명하는지}

## 목차

- [요약](#요약)
- [섹션1](#섹션1)
- [섹션2](#섹션2)
- [트러블슈팅](#트러블슈팅)
- [FAQ](#faq)

---

## 요약

{주요 명령 + 주요 절차 + 주요 개념}

---

## 섹션1

{내용}

---

## 섹션2

{내용}

---

## 트러블슈팅

{문제 해결}

---

## FAQ

{자주 묻는 질문}
```

### 필수 섹션

모든 가이드 문서는 다음 섹션을 포함해야 합니다:

| 섹션 | 위치 | 설명 |
|------|------|------|
| 제목 | 최상단 | `# {제목}` 형식 |
| 한 줄 설명 | 제목 바로 아래 | 문서의 목적을 한 문장으로 설명 |
| 목차 | 설명 다음 | 모든 `##` 레벨 섹션 링크 |
| 요약 | 목차 다음 | 주요 명령 + 주요 절차 + 주요 개념 |
| 트러블슈팅 | 끝에서 2번째 | 흔한 문제와 해결 방법 |
| FAQ | 마지막 | 자주 묻는 질문 (3~10개) |

### 선택 섹션

| 섹션 | 사용 시기 |
|------|----------|
| 개요 | 배경 설명이 필요할 때 |
| 참고 문서 | 외부/내부 참조가 있을 때 |




## 필수 섹션

### 1. 제목과 한 줄 설명

```markdown
# 단위 테스트 가이드

이 문서는 Functorium 프로젝트의 단위 테스트 작성 규칙과 패턴을 설명합니다.
```

**규칙:**
- 제목: `# {주제}` 형식
- 한 줄 설명: 제목 바로 아래, 빈 줄로 구분
- 간결하고 명확하게 작성

### 2. 목차

```markdown
## 목차

- [요약](#요약)
- [기본 명령어](#기본-명령어)
- [트러블슈팅](#트러블슈팅)
- [FAQ](#faq)
```

**규칙:**
- 모든 `##` 레벨 섹션을 포함
- 앵커는 소문자, 공백은 하이픈으로 변환
- 순서는 문서 구조를 따름

### 3. 요약

요약 섹션은 문서의 핵심 내용을 빠르게 참조할 수 있도록 합니다.

3개 하위 섹션으로 구성:

| 섹션 | 설명 |
|------|------|
| 주요 명령 | 바로 실행할 수 있는 명령어 (빠른 참조) |
| 주요 절차 | 단계별 워크플로우 |
| 주요 개념 | 핵심 원칙과 이론 |

### 4. 트러블슈팅

```markdown
## 트러블슈팅

### 커밋 메시지를 잘못 작성했을 때

**원인:** 설명

**해결:**
```bash
git commit --amend -m "올바른 메시지"
```
```

**규칙:**
- 문제 상황을 `###` 제목으로 명시
- 해결 방법을 코드 블록과 함께 제공
- 주의사항은 인용문(`>`)으로 강조

### 5. FAQ

```markdown
## FAQ

### Q1. 명령어 A와 명령어 B의 차이점은 무엇인가요?

| 명령어 | 동작 |
|--------|------|
| 명령어 A | 설명 A |
| 명령어 B | 설명 B |
```

**규칙:**
- 질문: `### Q{번호}. {질문}?` 형식
- 답변: 질문 바로 아래, 표/리스트/코드 블록 활용
- 3~10개 질문 권장




## 형식 규칙

### 섹션 구분

각 `##` 레벨 섹션 사이에 `---` 구분선을 추가합니다.

```markdown
## 섹션 1

내용...




## 섹션 2

내용...
```

### 표 (Table)

명령어, 옵션, 비교 정보에 사용합니다.

```markdown
| 명령어 | 설명 | 예시 |
|--------|------|------|
| `git status` | 작업 상태 확인 | `git status` |
```

**규칙:**
- 헤더와 데이터를 명확히 구분
- 코드는 백틱으로 감싸기
- 좌측 정렬 사용

### 강조

```markdown
**굵게** - 중요한 키워드, 강조
*기울임* - 부연 설명
`코드` - 명령어, 파일명, 변수명
```

### 인용

중요한 주의사항이나 팁을 강조할 때 사용합니다.

```markdown
> **주의**: 이 명령어는 되돌릴 수 없습니다.
> **팁**: 이 방법이 더 효율적입니다.
> **참고**: 추가 정보입니다.
```




## 코드 블록 작성

### 언어 명시

항상 코드 블록에 언어를 명시합니다.

````markdown
```bash
# Shell 명령어
git status
```

```csharp
// C# 코드
var result = await sut.Handle(request);
```

```json
{
  "key": "value"
}
```

```xml
<PropertyGroup>
  <IsTestProject>true</IsTestProject>
</PropertyGroup>
```
````

### 주석 스타일

사용자가 이해하기 쉽도록 주석을 추가합니다.

```bash
# 1. 변경사항 확인
git status

# 2. 파일 스테이징
git add .

# 3. 커밋
git commit -m "feat: 새 기능 추가"
```

### 좋은/나쁜 예시 구분

```csharp
// 올바른 예시
actual.IsSucc.ShouldBeTrue();

// 잘못된 예시
Assert.True(actual.IsSucc);
```

## 트러블슈팅

### 목차 링크가 작동하지 않을 때

**원인**: 앵커가 잘못되었습니다.

**해결**:
- 섹션 제목을 소문자로 변환
- 공백을 하이픈(`-`)으로 변경
- 특수문자 제거

```markdown
## 기본 명령어  → #기본-명령어
## FAQ         → #faq
## Git 설치    → #git-설치
```

### 코드 블록이 제대로 표시되지 않을 때

**원인**: 언어 명시를 빠뜨렸거나 백틱이 부족합니다.

**해결**: 반드시 세 개의 백틱과 언어를 명시합니다.

### 표가 깨져 보일 때

**해결**:
- 각 행의 열 개수가 동일한지 확인
- 헤더와 구분선이 있는지 확인

```markdown
| 열1 | 열2 | 열3 |
|-----|-----|-----|
| 값1 | 값2 | 값3 |
```




## FAQ

### Q1. 새 가이드 문서는 어떻게 시작하나요?

템플릿 섹션의 기본 템플릿을 복사하여 시작하세요. 주제에 맞게 섹션을 추가/제거합니다.

### Q2. 파일명은 어떻게 정하나요?

`{번호}-{주제}.md` 형식을 사용합니다:
- `15a-unit-testing.md`
- `A01-vscode-debugging.md`

번호는 카테고리별로 배정합니다. 하위 주제는 알파벳 접미사를 사용합니다 (예: `15b-integration-testing.md`).

### Q3. FAQ는 몇 개가 적당한가요?

3~10개를 권장합니다:
- 최소 3개: 기본적인 질문
- 권장 5~7개: 다양한 질문
- 최대 10개: 상세한 가이드

### Q5. 코드 블록에는 항상 언어를 명시해야 하나요?

네, 항상 명시하세요. `bash`, `csharp`, `json`, `xml`, `markdown` 등을 사용합니다.

### Q6. 표와 리스트 중 어느 것을 사용해야 하나요?

| 사용 | 형식 |
|------|------|
| 비교/매핑 정보 | 표 (Table) |
| 단순 나열 | 리스트 (List) |
| 순차적 절차 | 번호 리스트 |

### Q7. 다른 가이드 문서를 참조하려면 어떻게 하나요?

상대 경로로 링크하세요:

```markdown
[단위 테스트 가이드](./15a-unit-testing.md)를 참조하세요.

자세한 내용은 [에러 시스템](./08a-error-system.md#섹션)을 확인하세요.
```
