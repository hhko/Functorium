# Functorium Release v1.0.0-alpha.3

## Overview

Functorium v1.0.0-alpha.3 focuses on **IRepository redesign** and **EF Core performance optimization**. The repository interface has been reorganized into four clear responsibility groups--Write Single, Write Batch, Read, and Specification--while batch operations now bypass Change Tracker for significantly better throughput. A new `[GenerateSetters]` Source Generator eliminates the boilerplate required for EF Core `ExecuteUpdateAsync` adoption.

**Key Features**:

- **IRepository Redesign**: Interface reorganized into Write Single / Write Batch / Read / Specification groups with Specification-based `Exists`, `Count`, `DeleteBy` operations
- **EF Core Performance Optimization**: `Update` and `UpdateRange` now use `ExecuteUpdateAsync` to bypass Change Tracker; `CreateRange` supports chunked bulk inserts for large datasets
- **[GenerateSetters] Source Generator**: Automatically generates `ApplySetters` for EF Core models, eliminating manual `SetProperty` boilerplate

## Breaking Changes

### 1. CreateRange/UpdateRange Return Type Changed

Batch write operations now return the number of affected rows (`int`) instead of the collection of aggregates (`Seq<TAggregate>`). This aligns with the principle that batch callers already hold references to their aggregates and need only a success/count signal.

**Before (v1.0.0-alpha.2)**:
```csharp
// Batch operations returned the aggregate collection
FinT<IO, Seq<TAggregate>> CreateRange(IReadOnlyList<TAggregate> aggregates);
FinT<IO, Seq<TAggregate>> UpdateRange(IReadOnlyList<TAggregate> aggregates);

// Caller usage
var created = await repository.CreateRange(orders).RunAsync();
Seq<Order> orders = created.ThrowIfFail();  // Seq<Order>
```

**After (v1.0.0-alpha.3)**:
```csharp
// Batch operations now return affected count
FinT<IO, int> CreateRange(IReadOnlyList<TAggregate> aggregates);
FinT<IO, int> UpdateRange(IReadOnlyList<TAggregate> aggregates);

// Caller usage
var created = await repository.CreateRange(orders).RunAsync();
int count = created.ThrowIfFail();  // int (number of created/updated rows)
```

**Migration Guide**:
1. Update all `CreateRange` / `UpdateRange` call sites that use the return value as `Seq<TAggregate>` to expect `int` instead
2. If you need the aggregate collection after a batch operation, retain the input list directly
3. Custom repository subclasses overriding `CreateRange` or `UpdateRange` must update their return types

### 2. New Abstract Method `BuildSetters` Required in EfCoreRepositoryBase Subclasses

`EfCoreRepositoryBase` now requires all subclasses to implement the `BuildSetters` abstract method. This method provides the `SetProperty` mappings used by `ExecuteUpdateAsync` for the `Update` and `UpdateRange` operations.

**Before (v1.0.0-alpha.2)**:
```csharp
public class OrderRepository : EfCoreRepositoryBase<Order, OrderId, OrderModel>
{
    // Only ToDomain and ToModel were required
    protected override Order ToDomain(OrderModel model) => ...;
    protected override OrderModel ToModel(Order aggregate) => ...;
}
```

**After (v1.0.0-alpha.3)**:
```csharp
public class OrderRepository : EfCoreRepositoryBase<Order, OrderId, OrderModel>
{
    protected override Order ToDomain(OrderModel model) => ...;
    protected override OrderModel ToModel(Order aggregate) => ...;

    // New: required for ExecuteUpdateAsync
    protected override void BuildSetters(
        UpdateSettersBuilder<OrderModel> setters, OrderModel model)
        => OrderModel.ApplySetters(setters, model);  // Generated by [GenerateSetters]
}
```

**Migration Guide**:
1. Add `[GenerateSetters]` attribute to your EF Core model class and make it `partial`
2. Implement the `BuildSetters` method in your repository subclass by delegating to the generated `ApplySetters`
3. Add a reference to `Functorium.SourceGenerators` in your project

### 3. ExistsBySpec Renamed to Exists and Promoted to Public

The `protected ExistsBySpec` method on `EfCoreRepositoryBase` has been renamed to `Exists` and promoted to `public virtual`, now implementing the new `IRepository.Exists` contract.

**Before (v1.0.0-alpha.2)**:
```csharp
// Protected method - only usable inside repository subclass
protected FinT<IO, bool> ExistsBySpec(Specification<TAggregate> spec)
```

**After (v1.0.0-alpha.3)**:
```csharp
// Public method on IRepository interface
public virtual FinT<IO, bool> Exists(Specification<TAggregate> spec)
```

**Migration Guide**:
1. Rename all `ExistsBySpec` calls to `Exists`
2. If you were calling `ExistsBySpec` from a repository subclass, the method is now available on the `IRepository` interface directly

## New Features

### Functorium Library

#### 1. IRepository Interface Reorganization

The `IRepository<TAggregate, TId>` interface has been redesigned with four clearly separated responsibility groups. Each group follows a consistent return type convention: single-write operations return the aggregate for LINQ monad composition, batch-write operations return the affected row count, and specification operations execute as single SQL statements.

```csharp
public interface IRepository<TAggregate, TId> : IObservablePort
    where TAggregate : AggregateRoot<TId>
    where TId : struct, IEntityId<TId>
{
    // -- Write: Single --
    FinT<IO, TAggregate> Create(TAggregate aggregate);
    FinT<IO, TAggregate> Update(TAggregate aggregate);
    FinT<IO, int> Delete(TId id);

    // -- Write: Batch --
    FinT<IO, int> CreateRange(IReadOnlyList<TAggregate> aggregates);
    FinT<IO, int> UpdateRange(IReadOnlyList<TAggregate> aggregates);
    FinT<IO, int> DeleteRange(IReadOnlyList<TId> ids);

    // -- Read --
    FinT<IO, TAggregate> GetById(TId id);
    FinT<IO, Seq<TAggregate>> GetByIds(IReadOnlyList<TId> ids);

    // -- Specification --
    FinT<IO, bool> Exists(Specification<TAggregate> spec);
    FinT<IO, int> Count(Specification<TAggregate> spec);
    FinT<IO, int> DeleteBy(Specification<TAggregate> spec);
}
```

**Why this matters:**
- Clear separation of single vs. batch vs. read vs. specification responsibilities reduces cognitive load when navigating the interface
- Batch operations returning `int` instead of `Seq<TAggregate>` avoids materializing large aggregate collections that the caller already holds
- Specification-based `Exists`, `Count`, `DeleteBy` each execute as a single SQL statement (no client-side filtering), enabling efficient aggregate-level queries directly through the repository
- `UpdateBy(Specification, SetPropertyCalls)` is intentionally excluded from the interface to honor the dependency direction rule (EF Core types belong in the Adapter layer)

#### 2. Specification-Based Repository Operations: Exists, Count, DeleteBy

Three new Specification-based methods are now part of `IRepository`, enabling aggregate-level conditional queries that translate to single SQL statements. Both `EfCoreRepositoryBase` and `InMemoryRepositoryBase` provide implementations.

```csharp
// Check if any active orders exist for a customer (1 SQL: SELECT EXISTS)
var spec = new ActiveOrdersByCustomerSpec(customerId);
FinT<IO, bool> exists = repository.Exists(spec);

// Count matching aggregates (1 SQL: SELECT COUNT)
FinT<IO, int> count = repository.Count(spec);

// Bulk delete by specification (1 SQL: DELETE WHERE)
var expiredSpec = new ExpiredOrdersSpec(cutoffDate);
FinT<IO, int> deletedCount = repository.DeleteBy(expiredSpec);
```

**Why this matters:**
- Without these methods, checking existence or counting required loading all matching aggregates into memory, then filtering client-side
- Each operation compiles to a single SQL statement through EF Core's Expression translation, minimizing database round trips
- `DeleteBy` uses `ExecuteDeleteAsync` to bypass Change Tracker entirely, handling bulk deletes without loading entities
- Specification reuse: the same `Specification<TAggregate>` class works across `Exists`, `Count`, `DeleteBy`, and existing query infrastructure

### Functorium.Adapters Library

#### 3. EF Core Update Performance: ExecuteUpdateAsync Adoption

`Update` and `UpdateRange` in `EfCoreRepositoryBase` now use EF Core's `ExecuteUpdateAsync` instead of the traditional `FindAsync` + `SetValues` pattern. This eliminates the SELECT round trip that Change Tracker-based updates required.

```csharp
// Before: 2 round trips (SELECT + UPDATE)
// var existing = await DbSet.FindAsync(id);
// DbContext.Entry(existing).CurrentValues.SetValues(updated);

// After: 1 round trip (UPDATE WHERE Id = @id)
var model = ToModel(aggregate);
int affected = await DbSet
    .Where(ByIdPredicate(aggregate.Id))
    .ExecuteUpdateAsync(s => BuildSetters(s, model));
```

**Why this matters:**
- `Update` drops from 2 database round trips to 1 (no prior SELECT needed), halving latency for single-entity updates
- `UpdateRange` iterates `ExecuteUpdateAsync` per aggregate instead of loading all entities via Change Tracker, avoiding the N+1 SELECT problem
- Change Tracker bypass eliminates memory pressure from tracking modified entities in large batch scenarios
- The `protected UpdateBy(Specification, Action<UpdateSettersBuilder>)` method enables subclasses to implement domain-specific conditional bulk updates (e.g., "deactivate all expired subscriptions") as a single SQL UPDATE

#### 4. CreateRange Chunked Bulk Insert for Large Datasets

`CreateRange` now detects large input sets and automatically splits them into chunks, calling `SaveChangesAsync` + `ChangeTracker.Clear()` per chunk to prevent out-of-memory conditions.

```csharp
// Small batches (<= IdBatchSize): single AddRange + SaveChanges
// Large batches (> IdBatchSize): chunked processing

// EfCoreRepositoryBase automatically handles:
//   foreach (var chunk in aggregates.Chunk(IdBatchSize))
//   {
//       DbSet.AddRange(chunk.Select(ToModel));
//       await DbContext.SaveChangesAsync();
//       DbContext.ChangeTracker.Clear();
//   }

// Usage is unchanged:
FinT<IO, int> created = repository.CreateRange(largeOrderList);
```

**Why this matters:**
- Inserting tens of thousands of entities in a single `AddRange` + `SaveChangesAsync` call causes Change Tracker memory to grow unboundedly, leading to OOM in production
- Chunked processing with `ChangeTracker.Clear()` keeps memory usage constant regardless of batch size
- Small batches (below `IdBatchSize`) take the fast synchronous path with no chunking overhead
- This must be used within a `UsecaseTransactionPipeline` context to ensure atomicity across chunks; without a wrapping transaction, partial inserts cannot be rolled back

#### 5. [GenerateSetters] Source Generator

The new `[GenerateSetters]` attribute on EF Core model classes triggers a Source Generator that produces an `ApplySetters` static method, mapping each settable property to a `SetProperty` call for `ExecuteUpdateAsync`.

```csharp
using Functorium.Adapters.SourceGenerators;

[GenerateSetters]
public partial class OrderModel : IHasStringId
{
    public string Id { get; set; } = string.Empty;     // Auto-excluded (Id)
    public string Status { get; set; } = string.Empty;
    public decimal TotalAmount { get; set; }

    [SetterIgnore]
    public DateTime AuditTimestamp { get; set; }        // Manually excluded

    public ICollection<OrderItemModel> Items { get; set; } = [];  // Auto-excluded (navigation)
}

// Generated method (by Functorium.SourceGenerators):
// public static void ApplySetters(UpdateSettersBuilder<OrderModel> s, OrderModel m)
// {
//     s.SetProperty(x => x.Status, m.Status);
//     s.SetProperty(x => x.TotalAmount, m.TotalAmount);
// }

// Use in repository:
protected override void BuildSetters(
    UpdateSettersBuilder<OrderModel> setters, OrderModel model)
    => OrderModel.ApplySetters(setters, model);
```

**Why this matters:**
- Writing `SetProperty` calls manually for each model property is tedious and error-prone; adding a new column requires updating both the model and the repository
- The Source Generator keeps `ApplySetters` in sync with the model automatically -- add a property, and the setter mapping is generated at build time
- `Id` properties and navigation properties (`ICollection<T>`, `IList<T>`) are automatically excluded, preventing accidental primary key overwrites or navigation errors
- `[SetterIgnore]` provides fine-grained control for audit columns or computed fields that should not be included in updates

## API Changes

### IRepository Interface (Functorium)

```
Functorium.Domains.Repositories
└── IRepository<TAggregate, TId>
    ├── Write: Single
    │   ├── Create(TAggregate) -> FinT<IO, TAggregate>
    │   ├── Update(TAggregate) -> FinT<IO, TAggregate>
    │   └── Delete(TId) -> FinT<IO, int>
    ├── Write: Batch
    │   ├── CreateRange(IReadOnlyList<TAggregate>) -> FinT<IO, int>     [Changed: was Seq<TAggregate>]
    │   ├── UpdateRange(IReadOnlyList<TAggregate>) -> FinT<IO, int>     [Changed: was Seq<TAggregate>]
    │   └── DeleteRange(IReadOnlyList<TId>) -> FinT<IO, int>
    ├── Read
    │   ├── GetById(TId) -> FinT<IO, TAggregate>
    │   └── GetByIds(IReadOnlyList<TId>) -> FinT<IO, Seq<TAggregate>>
    └── Specification                                                    [New group]
        ├── Exists(Specification<TAggregate>) -> FinT<IO, bool>          [New]
        ├── Count(Specification<TAggregate>) -> FinT<IO, int>            [New]
        └── DeleteBy(Specification<TAggregate>) -> FinT<IO, int>         [New]
```

### EfCoreRepositoryBase New Members (Functorium.Adapters)

```
Functorium.Adapters.Repositories
└── EfCoreRepositoryBase<TAggregate, TId, TModel>
    ├── abstract void BuildSetters(UpdateSettersBuilder<TModel>, TModel)  [New: required]
    ├── virtual FinT<IO, bool> Exists(Specification<TAggregate>)          [New: was protected ExistsBySpec]
    ├── virtual FinT<IO, int> Count(Specification<TAggregate>)            [New]
    ├── virtual FinT<IO, int> DeleteBy(Specification<TAggregate>)         [New]
    └── protected FinT<IO, int> UpdateBy(Specification<TAggregate>, Action<UpdateSettersBuilder<TModel>>)  [New]
```

### Source Generator Attributes (Functorium.Adapters)

```
Functorium.Adapters.SourceGenerators

```
